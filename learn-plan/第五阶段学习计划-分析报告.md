第五阶段学习计划
从一台服务器到一个系统：引入对象存储
2026年6月 | 路线图阶段：第五阶段（第9~12周） | 难度：中高级
一、问题说明
1.1 问题描述
第四阶段结束后，你拥有了一套自动化部署的容器化应用：Nginx + Flask + MySQL + Docker Compose + GitHub Actions CI/CD。每次 git push 之后服务器自动更新。架构上你只使用了 ECS 和 RDS 两类云服务。
现实中云应用的数据不只是存数据库。用户上传的图片、应用生成的报表、系统的日志文件——这些非结构化数据需要专门的存储服务。直接存 ECS 硬盘上有这些问题：
•服务器磁盘有限：ECS 几十 GB，图片多了很快撑满
•扩容麻烦：磁盘满了要停机扩容、买新盘、挂载、迁移数据
•没有高可用：ECS 坏了磁盘损坏，图片全丢
•影响性能：大文件读写占用磁盘 IO 和带宽，拖慢 Flask 响应
•无法做 CDN 加速：图片从 ECS 单点传输，离用户越远越慢
第五阶段的核心任务：引入阿里云 OSS 对象存储，把图片从 ECS 硬盘迁移到云端。同时为应用加上基础防护（上传限制、频率限制、简单登录），让图床变成一个能拿得出手的项目。
1.2 学习背景与意义
OSS 是云原生应用中继计算和数据库之后的第三大基础服务。核心价值：无限容量（不必关心磁盘）、高可用（99.99999999% 可靠性）、流量直连（上传下载不经过你的服务器）。
核心价值：
•存算分离：图片不存服务器本地，服务器只存链接。ECS 管处理请求，OSS 管存文件——各司其职
•弹性扩容：一张图和一万张图对你来说没区别，阿里云自动扩缩容
•自带外部访问：OSS 上的图片自带公网 URL，浏览器可以直接访问，不需要服务器当图片快递员
•防数据丢失：OSS 自动做三份冗余备份（跨机房），ECS 磁盘坏了图片还在
•CDN 友好：OSS 天然对接 CDN，图片被分发到离用户最近的节点
和第四阶段完美衔接：Docker + Nginx + Flask + MySQL + CI/CD 五层基建全部复用。只是在现有架构上横向增加一个 OSS 云服务，不是推倒重来。
1.3 涉及的核心概念
以下是本阶段需要理解的核心概念，按学习顺序排列：
概念	在本阶段你需要知道的程度
对象存储 OSS	不存服务器硬盘，直接存云上。上传后自动返回公网 URL。类比：网盘 API 版——你只管调接口上传下载，硬盘、备份、扩容全由阿里云处理
Bucket（存储桶）	OSS 的顶层容器，类似于云上的一个文件夹。每个 Bucket 有独立的权限、地域、域名
AccessKey 认证	OSS 不走用户名密码，而是 AccessKey ID + Secret。类似 GitHub SSH 私钥，是程序访问云资源的身份证。绝不能写死在代码或提交 Git
Object（对象）	OSS 中存储的每个文件叫一个 Object。包含文件数据 + 元数据（文件名、大小、上传时间）+ 唯一 URL
multipart/form-data	文件上传的标准 HTTP 编码格式。浏览器用 FormData 对象包装文件，Flask 通过 request.files 接收
频率限制 (Rate Limiting)	防止恶意刷接口。同一 IP 短时间内请求超过阈值返回 429 Too Many Requests
Session 认证	最简单的用户认证方式。输密码 → Flask 校验 → 写入 cookie → 后续请求读 cookie。不需要数据库和 OAuth
内容安全	用户上传的不一定是图片。服务端必须校验文件类型（后缀 + MIME 白名单），不能信任用户发来的任何东西
二、解决方式
2.1 方案概述
总体思路：在第四阶段的 Docker Compose 架构上横向扩展——新增 OSS 作为图片存储，Flask 新增上传接口，前端新增上传页面。Nginx 反向代理、MySQL、CI/CD 流水线全部保持不动。整个过程约需 2~3 天。应用代码新增约 100 行，基础设施文件微调但不需要重写。
架构变化（虚线框表示容器，OSS 是云服务不容器化）：
┌──────────────────── 宿主机 (ECS) ────────────────────┐
│                                                       │
│  ┌─ 容器A (Nginx) ──┐  ┌─ 容器B (Flask) ──┐        │
│  │  :80               │  │  :5000             │        │
│  │  index.html        │  │  POST /api/upload ─┼──→ OSS
│  │  upload.html ──────┼──┼─ GET  /api/images ─┼──→ MySQL
│  │  nginx.conf        │  │                    │        │
│  └────────────────────┘  └────────────────────┘        │
└───────────────────────────────────────────────────────┘
注意：和第四阶段一样，云服务（OSS + MySQL）不放进容器。容器化只针对自己部署的 Nginx 和 Flask。
2.2 详细步骤
第 1 天：开通 OSS + 跑通上传流程
•在阿里云控制台开通 OSS，创建 Bucket（地域选和 ECS 同可用区，权限先设私有）
•获取 AccessKey：RAM 控制台 → 创建子用户 → 勾选 OpenAPI 调用 → 只赋 OSS 权限（最小权限原则）
•理解三个核心概念：Bucket（命名空间，全局唯一）、Object（一个文件一个对象）、Endpoint（OSS 访问入口 URL，分内网和外网）
•用 Python 脚本试 put_object_from_file：pip install oss2，上传一张本地图片到 Bucket
•写 Flask 上传接口：POST /api/upload，接收 request.files，用 oss2 SDK 上传，公网 URL 写入 MySQL
•写 Flask 列表接口：GET /api/images，从 MySQL 查询所有图片链接返回 JSON
•验收：curl -F "file=@测试图.png" http://127.0.0.1:5000/api/upload，返回 OSS URL，浏览器能打开
第 2 天：容器化 + 对接 Nginx
•更新 requirements.txt，加上 oss2 + flask-limiter
•更新 Dockerfile（基本不变，pip install 多装两个包）
•更新 nginx.conf：新增 location 返回上传页面，原有 location /api/ 照旧
•更新 app.py：新增 upload/images 路由，init_db() 加 images 表建表语句
•AccessKey 走环境变量 OSS_ACCESS_KEY_ID 和 OSS_ACCESS_KEY_SECRET，和数据库密码一样在 .env 管理
•docker-compose up -d --build，浏览器打开上传页面，拖拽图片上传
•验收：上传 → 看到 OSS 链接 → 浏览器打开可见图片 → 图片列表展示所有已上传图片
第 3 天：安全防护 + CI/CD 串联
•加上传大小限制：Flask 端 MAX_CONTENT_LENGTH = 5MB，前端也检查
•加文件类型检查：allowed_extensions = {jpg,jpeg,png,gif,webp}，拒绝其他后缀
•加频率限制：flask-limiter，上传接口限制每分钟 5 次
•加简单 Session 登录：密码存环境变量 UPLOAD_PASSWORD，未登录重定向到登录页
•更新 .env，把 OSS_KEY_ID、OSS_KEY_SECRET、UPLOAD_PASSWORD 加进去
•push master 触发 CI/CD，验证全自动部署
•验收：登录 → 上传 → 看到链接 → 新标签打开 → 超限被拒绝 → 未登录被拦截
2.3 核心原理分析
图片为什么不存服务器，要存 OSS？
对比维度	存 ECS 硬盘	存 OSS
容量	几十 GB，满了要加盘	无限容量，自动扩缩容
可靠性	磁盘损坏 = 图片丢失	99.99999999%，三副本跨机房
流量	上传下载走 ECS 带宽，高峰卡顿	上传下载直连 OSS，不占 ECS 带宽
访问速度	单点传输，离用户越远越慢	自带 CDN 加速，就近访问
成本	ECS 磁盘贵且浪费	按量付费，存多少付多少
扩容	停机买盘挂载迁移——运维噩梦	什么都不用做
 
AccessKey 为什么不能写在代码里？
AccessKey 等于你的阿里云支付密码。谁拿到它就能花你的钱。写进代码 = 推到 GitHub = 全世界可见。正确做法：存 .env → .gitignore 忽略 .env → 部署时环境变量注入。
 
存算分离到底是什么？
你的图床中，服务器只做两件事：处理上传请求（算），转发图片到 OSS（存）。图片的持久化存储、备份、分发全部由 OSS 负责——服务器不存文件。这就是存算分离：计算层和存储层解耦，各自独立扩缩容，互不拖累。

2.5 预计踩坑与应对
现象	可能原因	排查方法
OSS SDK 报 AccessDenied	AccessKey 没权限或填错；Bucket 策略不对	1. .env 检查 Key 完整性 2. RAM 控制台确认权限 3. Bucket 权限检查
上传成功但 URL 打不开	Bucket 权限是私有	确认 Bucket 读写权限；私有需生成签名 URL
大图片上传超时	Nginx proxy_read_timeout 默认 60s	nginx.conf 加 proxy_read_timeout 120s
上传后列表无新图片	POST 返回 201 但 commit 未生效	检查 INSERT 后是否有 conn.commit()
未登录也能上传	Session 校验遗漏	所有上传路由加 @login_required 装饰器；curl -b 无 cookie 测试
AccessKey 被提交 Git	手滑 git add . 把 .env 提交了	立即禁用泄露 Key；.gitignore 加 .env；代码只用 os.getenv()
频率限制不生效	flask-limiter 配置或 IP 获取不对	确认 Limiter 初始化；生产环境从 X-Forwarded-For 取真实 IP
