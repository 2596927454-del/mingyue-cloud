第三阶段学习计划
云数据库替代 SQLite，跑通最后一块拼图
2026年5月 | 路线图阶段：第三阶段（第5~6周） | 难度：初级偏中级


一、问题说明
1.1 问题描述
第二阶段结束后，你已经拥有一个完整的前后端分离个人网站：Nginx 托管静态简历页 + 反向代理转发 API 请求给 Flask + SQLite 存储留言数据。这条链路已经跑通，但 SQLite 是一个本地文件数据库——存在两个根本性问题：
数据绑死在服务器上：messages.db 文件在服务器本地磁盘里。服务器磁盘坏了、被误删了、想换一台服务器，数据就没了
不支持多服务器共享：未来想扩展（比如加一台服务器做负载均衡），两台服务器各有一个自己的 messages.db，A 服务器上的留言在 B 服务器上看不到
第三阶段要做的，就是把 SQLite 替换成云厂商提供的托管 MySQL 数据库服务。这是你前两个阶段架构中唯一还没碰的组件，也是"云计算"这个词里最核心的那一半——"计算"你已经在用了（ECS 服务器），"数据"还没有。
1.2 学习背景与意义
第三阶段的本质不是"学 MySQL 语法"，而是理解"云数据库"这个概念本身。

核心价值：
第一次用上"云服务"而不是"自己装软件"：Phase 2 的 Flask 是你手动安装配置的，Phase 3 的 MySQL 是云厂商帮你配好的——你只拿到一个连接地址。这是云计算的核心逻辑：把运维交给平台
理解数据库和文件的本质区别：SQLite 是文件，MySQL 是服务。文件只能本地读写，服务可以通过网络远程连接。这个区别决定了生产环境为什么必须用后者
完整链路上的最后一块拼图：前端 → Nginx 反向代理 → Flask 后端 → 云数据库，这四个角色全部就位后，你的架构已经是一套可以投产的最小化 Web 应用
安全性意识的升级：数据库密码不能硬编码在代码里（传到 GitHub 就泄露了），要用环境变量。云数据库也不能暴露公网，只能通过内网访问

和前两个阶段的串联关系：
这个阶段几乎不写新代码——前端 index.html 一行不动，Nginx 配置一行不动，Flask app.py 只改数据库连接那几行。本质上是在 Phase 2 搭好的架构框架上，把最后一个组件（SQLite）换成生产级方案（云 MySQL）。这就是"前后端分离"的甜头——换数据库，其他层完全无感。
1.3 涉及的核心概念
以下是本阶段需要理解的核心概念，按学习顺序排列：
概念	在本阶段你需要知道的程度
云数据库 (RDS/CloudDB)	云厂商提供的托管数据库服务。你不需要自己装 MySQL、配参数、做备份——云厂商帮你搞定。你只需要在控制台点几下，拿到一个"连接地址"
MySQL	目前最流行的开源关系型数据库。本阶段只需要知道：用 SQL 语句操作数据、和 SQLite 的 SQL 语法几乎一样、区别是 MySQL 是一个独立的网络服务而不是文件
PyMySQL	Python 连接 MySQL 的库，类似第二阶段用的 sqlite3 模块。用法差不多，区别是多了一个"连接参数"——要指定主机地址、端口、用户名、密码
内网连接	云数据库不暴露在公网上，只能被同一内网（VPC）下的 ECS 服务器访问。攻击者从公网扫不到你的数据库，安全性比暴露端口高一个级别
环境变量	把密码、地址这类敏感信息放在系统环境变量里，代码通过 os.getenv() 读取。这样做的好处是代码可以公开（push GitHub），密码不会泄露
数据库连接池	维护一组已建立好的数据库连接，请求来了直接复用，不用每次都重新连接。提高性能，也防止频繁建连/断连把数据库压垮
DDL / DML	SQL 语句的两大类。DDL（Data Definition Language）操作表结构——CREATE TABLE、ALTER TABLE；DML（Data Manipulation Language）操作数据本身——SELECT、INSERT、UPDATE、DELETE
二、解决方式
2.1 方案概述
总体思路：在云控制台开通 MySQL 实例（最低配，约 20~30 元/月，通常有新人免费额度）→ 创建数据库和表 → 修改 Flask 代码，把 sqlite3 换成 PyMySQL → 用环境变量管理数据库密码 → 重启 Flask 验证整条链路。整个过程约需 3~4 天，代码改动不超过 20 行。

架构变化（改动部分标粗）：

浏览器
  │
  ├─ GET /index.html ──▶ Nginx :80 ──▶ 静态文件 (不动)
  │
  └─ GET/POST /api/messages ──▶ Nginx :80 ──▶ proxy_pass ──▶ Flask :5000
                                                              │
                                                              ▼
                                                    【SQLite messages.db  ──▶  云 MySQL】
                                                      (本地文件)              (网络服务)

2.2 详细步骤
第 1 天：开通云数据库 MySQL 实例
登录云厂商控制台，找到"云数据库"或"RDS"产品入口
创建 MySQL 实例，配置选型：版本 MySQL 8.0，规格最低配（1核 1GB），存储 20GB，网络选择**和你的 ECS 同一个 VPC**（否则内网不通）
创建数据库账号（root 级别的用户名和密码），创建数据库名（如 guestbook）
在实例详情中复制"内网地址"（类似 mysql-xxx.mysql.rds.aliyuncs.com 或 10.x.x.x）
验收：在 ECS 服务器上用 mysql 客户端测试能否连上：mysql -h 内网地址 -u 用户名 -p

第 2 天：修改 Flask 代码，对接 MySQL
安装 PyMySQL：在虚拟环境中 pip install pymysql
修改 app.py 的数据库连接部分：把 sqlite3.connect("messages.db") 替换为 pymysql.connect(host=..., user=..., password=..., database=...)
修改 SQL 语句中的参数占位符：SQLite 用 ?，MySQL 用 %s（如 INSERT INTO messages (nickname, content, created_at) VALUES (%s, %s, %s)）
修改建表语句：SQLite 的 INTEGER PRIMARY KEY AUTOINCREMENT 改为 MySQL 的 INT AUTO_INCREMENT PRIMARY KEY；TEXT 改为 VARCHAR 或 TEXT
把数据库连接信息（host、user、password、database）改为从环境变量读取，不要硬编码在代码里
验收：本地或服务器上启动 Flask，curl 测试 GET 和 POST 接口正常

第 3 天：部署上线 + 安全加固
在 ECS 服务器上设置环境变量（export DB_HOST=xxx、export DB_USER=xxx 等），写入 ~/.bashrc 或 systemd service 文件使其持久化
云数据库安全加固：白名单只允许 ECS 的内网 IP 访问（不是 0.0.0.0/0）
ECS 安全组确认：不需要放行 MySQL 3306 端口到公网（走内网连接，不经过公网安全组）
重启 Flask，测试留言板功能：提交留言 → 检查云数据库控制台的数据量统计是否增长
验收：通过域名访问，留言板功能完全正常，数据存在云 MySQL 而非服务器本地文件

第 4 天（可选）：数据迁移 + 理解连接池
把 SQLite 中已有的留言数据导出，导入到 MySQL（体会"数据迁移"的概念）
了解连接池的基本概念：Flask 每次请求都新建连接的话性能很差，用 DBUtils 或 SQLAlchemy 维护连接池
理解 flask + g 对象管理数据库连接的标准模式（每个请求一个连接，请求结束归还连接池）
总结：对比 SQLite 文件和云 MySQL 的优缺点，形成对"什么时候用文件数据库，什么时候用网络数据库"的判断力
2.3 核心原理分析
SQLite 和 MySQL 的本质区别是什么？

SQLite 是一个"库"（Library），MySQL 是一个"服务"（Service）。这是两者最底层的区别，决定了所有差异：
对比维度	SQLite	MySQL
运行方式	嵌入在 Python 进程里，import sqlite3 就调用了	独立进程，监听 3306 端口，Flask 通过网络连接它
数据存储	一个 .db 文件，和代码放在一起	数据文件由 MySQL 服务管理，你不需要关心存在哪个目录
连接方式	本地文件读写，无网络开销	TCP 网络连接（内网），有连接建立/维护成本
并发能力	WAL 模式支持有限并发，写操作串行	真正的多用户并发，支持同时大量读写
多服务共享	不行，文件在本地磁盘，其他服务器读不到	可以，通过网络连接，多台 Flask 连同一个 MySQL
运维	零运维，文件在就数据在	云厂商托管运维（备份、监控、升级），你只管用
适用场景	嵌入式、移动端、开发测试、低并发单机	生产环境 Web 应用、多服务器、高并发

为什么云数据库要通过内网访问而不是公网？
安全性：公网 IP 暴露 3306 端口 = 全世界都能尝试连接你的数据库。通过内网，只有你自己的 ECS 能访问
性能：内网延迟 < 1ms，公网延迟 5~50ms。每条 SQL 请求差几十毫秒，积累起来差距巨大
流量成本：内网流量免费，公网流量按量计费
这就是"云上架构"的核心设计原则——计算层（ECS）可以暴露公网，数据层（RDS）永远只有内网

环境变量解决了什么问题？
你在 Phase 2 的 app.py 里，数据库路径是硬编码的：DB = "messages.db"。这在 SQLite 时代没问题——就是个本地文件名，不敏感。但 MySQL 的连接信息完全不同：
数据库密码：泄露了 = 别人能删你数据
数据库地址：暴露了 = 别人知道你的数据库在哪（虽然内网地址外网连不上，但不应公开）

环境变量的做法：代码里只写 os.getenv("DB_PASSWORD")，真正的密码存在服务器的环境变量里。代码 push 到 GitHub 不会泄露密码，不同的环境（开发/生产）可以配不同的密码，换数据库密码时只改环境变量不用改代码。这是 12-Factor App 方法论中的第三项——"配置与代码分离"。

连接池是什么？
不用连接池时，每次请求都要：TCP 三次握手 → MySQL 认证 → 执行 SQL → 关闭连接。一条留言提交光建立和关闭连接就花 3~50ms，高并发时还可能导致 MySQL 连接数耗尽。
用了连接池后：Flask 启动时预先建好几个连接，请求来了直接拿一个现成的用，用完还回去（不断开）。省掉了重复建连的开销，也限制了最大连接数防止打爆数据库。这就是"池化"思想——和线程池、HTTP 连接池是同一个道理。最小化连接池设置通常为 2~5 个连接，足够留言板这种小项目使用。

2.5 预计踩坑与应对
现象	可能原因	排查方法
pip install pymysql 报错	不在虚拟环境中；需要先激活 venv	source venv/bin/activate 后再安装
Flask 报 Cant connect to MySQL server	MySQL 实例和 ECS 不在同一个 VPC；白名单没放行 ECS 内网 IP；3306 端口没开	1. 确认 VPC 是否一致 2. 检查 MySQL 白名单 3. 云数据库控制台确认内网地址是否正确
Flask 报 Access denied for user	用户名或密码错了	在 ECS 上用 mysql 命令行客户端用同样的用户名密码测试连接
Flask 报 Unknown database	数据库名没创建	在 MySQL 控制台或客户端执行 CREATE DATABASE guestbook
留言提交成功但刷新后看不到	插入操作有 bug（连接没 commit）	pymysql 默认开启事务，插入后需要 conn.commit() 或在 connect 时设置 autocommit=True
Flask 启动正常但 curl 报 500	SQL 语法不兼容——SQLite 和 MySQL 的占位符和自增主键语法不同	检查 ? 是否都改成了 %s，INTEGER PRIMARY KEY AUTOINCREMENT 是否改成了 INT AUTO_INCREMENT PRIMARY KEY
云数据库费用超出预期	选了过高的规格；开启了公网访问产生流量费	用最低配、关闭公网访问、设置费用告警
环境变量重启后丢失	export 命令只在当前 session 有效	写入 ~/.bashrc、/etc/environment 或使用 systemd service 文件设置 Environment
