告别手工部署，学会标准化的交付方式

一、问题说明
1.1 问题描述
前三个阶段结束后，你拥有了一套完整的应用架构：Nginx 托管静态简历 + 反向代理转发 API 给 Flask + 云 MySQL 存储数据。但整个部署过程是纯手动的：
每次更新代码，要手动上传文件到服务器、手动重启 Flask、手动 reload Nginx
服务器环境依赖你的记忆：Python 版本、虚拟环境路径、装了哪些包、ngix.conf 改了哪里——换一台服务器或者重装系统，全部重新配一遍
没有版本化：你不知道当前跑的是哪个版本，回滚意味着"把旧代码手动拷回去"
进程管理脆弱：Flask 后台跑、kill 杀、WAL 锁文件、端口占用——你踩过的每一个坑换个环境都可能重演

第四阶段的核心任务：把你的留言板项目用 Docker 容器化，实现"一次构建，到处运行"。一条命令就能在任何服务器上拉起整套环境，不再依赖手动配置。
1.2 学习背景与意义
Docker 是云原生世界的第一个入口。它的核心价值只有一个：消除"在我电脑上能跑啊"的问题。

核心价值：
环境即代码：所有依赖（Python 版本、pip 包、Nginx 配置、系统库）全部写在 Dockerfile 里，一个纯文本文件描述完整运行环境。新服务器只需要装 Docker，不需要装 Python、不需要配 venv、不需要装 Nginx
一次构建，到处运行：在你本地打好镜像，推到仓库，服务器 docker pull 下来直接跑——不需要在服务器上 pip install 任何东西
隔离性：每个容器有自己的文件系统、网络、进程空间。Flask 的 5000 端口只在容器内部可见，不会和宿主机的 5000 冲突。你之前被 PID 残留和端口占用折磨过，Docker 用容器边界从根本上隔离了这些问题
版本化与回滚：每个镜像有 tag（比如 v1.0、v2.1），回滚就是 docker run 旧 tag，比手动拷代码快一百倍
和前三阶段完美衔接：应用代码（index.html、app.py、nginx.conf）一行不改，只是外面套了一层容器封装
1.3 涉及的核心概念
以下是本阶段需要理解的核心概念，按学习顺序排列：
概念	在本阶段你需要知道的程度
Docker 镜像 (Image)	一个只读的模板，包含了运行应用所需的所有文件（代码、依赖、系统库、环境变量）。类比：烤蛋糕的模具——模具做一次，蛋糕可以无限制复制
Docker 容器 (Container)	镜像的运行实例，每个容器互相隔离。类比：用同一个模具烤出来的一个个独立蛋糕，吃一个不影响其他的
Dockerfile	一个纯文本文件，描述了"怎么构建这个镜像"的步骤。从基础镜像开始（FROM），安装依赖（RUN），拷贝代码（COPY），指定启动命令（CMD）
Docker Compose	管理多容器应用的工具。用 YAML 文件定义"这个应用由哪些容器组成"（Nginx + Flask），一条 docker compose up 全部启动
镜像仓库 (Registry)	存镜像的地方。Docker Hub 是公共的，云厂商也有私有的（阿里云容器镜像服务）。相当于 GitHub for 代码 ↔ Registry for 镜像
CI/CD	持续集成 / 持续部署。代码 push 到 GitHub 后自动构建镜像，自动部署到服务器。本阶段只需了解概念，代码提交到 GitHub 即可，自动部署部分作为可选项
宿主机 vs 容器	宿主机就是你的 ECS 服务器（装了 Docker 的那台）。容器是跑在宿主机之上的隔离进程，对容器内的应用来说"全世界就是容器"——它看不到宿主机上的其他进程和文件
二、解决方式
2.1 方案概述
总体思路：为留言板的每个组件分别写 Dockerfile（Flask 一个，Nginx一个），用 Docker Compose 把两者编排在一起。整个过程约需 3~4 天。应用代码不动，但部署方式从"SSH 进去手动敲命令"变成"docker compose up 一键拉起。""

架构变化（虚线框表示 Docker 容器）：

┌──────────────── 宿主机 (ECS) ────────────────┐
│                                               │
│  ┌─ 容器 A (Nginx) ──┐  ┌─ 容器 B (Flask) ──┐│
│  │  :80               │  │  :5000             ││  云 MySQL
│  │  index.html        │  │  app.py            ││  (独立服务,
│  │  nginx.conf ───────┼──┼─▶ proxy_pass ─────┼│   不容器化)
│  └────────────────────┘  └────────────────────┘│
│                                               │
└───────────────────────────────────────────────┘

注意：云 MySQL 不放进容器——它是托管服务，独立于 ECS 存在。容器化只针对你手动管理的组件（Nginx 和 Flask）。这也是云架构的常见实践：SaaS/托管服务不容器化，自己部署的应用才容器化。
2.2 详细步骤
第 1 天：安装 Docker + 理解镜像和容器
在 ECS 服务器上安装 Docker：apt install docker.io -y && systemctl enable --now docker
跑通 hello-world：docker run hello-world（Docker 会自动从 Docker Hub 下载镜像并运行）
理解基本流程：docker pull 拉镜像 → docker run 启动容器 → docker ps 看运行中的容器 → docker stop 停容器
理解 docker run 的常用参数：-d（后台运行）、-p 宿主机端口:容器端口（端口映射）、-v 宿主机路径:容器路径（挂载卷）、--name（给容器取名）
验收：能用 docker ps 看到运行中的容器，能 docker stop + docker rm 清理容器

第 2 天：写 Dockerfile，打包 Flask 应用
在项目目录下创建 requirements.txt（flask + pymysql + dbutils），这是 Docker 构建时安装依赖的依据
编写 Dockerfile：FROM python 基础镜像 → COPY 代码 → RUN pip install → CMD 启动 Flask
docker build -t guestbook-flask . 构建镜像
docker run -d -p 5000:5000 --env-file .env guestbook-flask 启动容器，用 .env 文件传数据库连接信息
验收：curl http://127.0.0.1:5000/api/messages 正常返回 JSON

第 3 天：Docker Compose 编排 Nginx + Flask
编写 docker-compose.yml：定义 flask 服务和 nginx 服务，配置端口映射、网络互通、环境变量
Flask 容器不映射端口到宿主机（只对内网暴露，走 Nginx 转发）——和原来 Flask 监听 127.0.0.1 的逻辑一样，只是这次隔离靠 Docker 网络而不是 127.0.0.1
Nginx 容器映射 80 和 443 到宿主机，nginx.conf 里的 proxy_pass 指向 Flask 容器名（Docker Compose 内部 DNS 自动解析）
用 .env 文件统一管理数据库密码等敏感信息，不写死在 docker-compose.yml 里
验收：docker compose up -d 一键启动，浏览器访问域名，留言板功能完全正常

第 4 天（选做）：GitHub + CI/CD 入门
在 GitHub 上把项目代码（包括 Dockerfile、docker-compose.yml）推上去
了解 CI/CD 的基本概念：代码 push → GitHub Actions 触发 → 自动构建镜像 → 自动部署到服务器
可选项：配置 GitHub Actions 工作流，实现 git push 后服务器自动拉新镜像并重启容器
验收：本地 git push，服务器自动更新到最新版本
2.3 核心原理分析
Docker 和之前手动部署的本质区别是什么？

对比维度	手动部署（Phase 2/3）	Docker 部署（Phase 4）
环境准备	SSH 上去装 Python、创建 venv、pip install、装 Nginx、改 nginx.conf	服务器只需要装 Docker，其余全部在 Dockerfile 里
启动方式	手动 source venv/bin/activate && python app.py &	docker compose up -d 一条命令
依赖管理	依赖安装在系统上，多个项目可能冲突	每个容器有独立的依赖环境，互不干扰
进程管理	手动 kill、查端口、清理 PID、清理 WAL 锁文件	docker compose stop/restart，容器边界隔离，不存在端口冲突
回滚	把旧代码手动 scp 上去覆盖	docker run 旧镜像 tag，秒级回滚
环境迁移	换服务器 = 重头配一遍	新服务器装 Docker，docker compose up，完全相同的结果
可复现性	"在我这能跑"问题频发	Dockerfile 描述精确环境，任何人构建结果一致

Docker 是如何做到隔离的？
Docker 利用了 Linux 内核的两个技术：Namespace 和 Cgroup。

Namespace（命名空间隔离）：让容器内的进程以为自己独占一台机器。PID namespace 让容器只能看到自己内部的进程（Flask 容器里看不到宿主机的其他进程）；Network namespace 让容器有自己独立的 IP 地址和端口空间（Flask 的 5000 只在容器内有效，和宿主机的 5000 是不同的两个"世界"）；Mount namespace 让容器有自己独立的文件系统（容器内 /app/ 指向代码，看不到宿主机的 /root/ 目录）。

Cgroup（资源限流）：防止一个容器把宿主机的 CPU/内存吃光。可以限制容器只能用 1 个 CPU 核、512MB 内存——这对于生产环境非常重要，防止某个服务出 bug 拖垮整台服务器。

类比：Namespace 是"分房间"——每个住客（容器）有自己的房间，互相看不见。Cgroup 是"限水限电"——每个房间最多用多少水电，超了就不给了。

Docker Compose 的网络是怎么互通的？
当你写 docker-compose.yml 时，Compose 自动创建一个虚拟网络，把 flask 和 nginx 两个容器接在同一个网络上。每个容器可以通过服务名（如 flask）作为 DNS 域名访问另一个容器：

原来 nginx.conf 里写的是：proxy_pass http://127.0.0.1:5000;（通过宿主机 loopback 找 Flask）
Docker 化后变成：proxy_pass http://flask:5000;（通过 Compose 内部 DNS 找到 flask 容器）

127.0.0.1 是宿主机环回地址，容器内使用 127.0.0.1 只能指向容器自己。flask 是 Compose 给容器分配的主机名，Nginx 容器通过这个名称访问 Flask 容器的 5000 端口。这就是服务发现的最简单形式。

为什么云 MySQL 不放进容器？
它是一个托管服务，独立于 ECS 存在。把它容器化等于在托管服务外面又套了一层——多此一举且引入不必要的复杂度
容器是"可以被销毁和重建"的（docker rm + docker run），数据应该放在容器外面的持久化存储上。云 MySQL 本身就是持久化存储——放在外面是合理的
这是云计算架构的通用原则：SaaS/托管服务（云数据库、对象存储、CDN）保持原样，自己开发的无状态应用（Flask、Nginx）才用容器

2.5 预计踩坑与应对
现象	可能原因	排查方法
docker: command not found	Docker 没装或装了但没配用户组	apt install docker.io -y && usermod -aG docker $USER，重新登录
docker build 很慢或卡住	基础镜像很大；网络问题 Docker Hub 拉镜像慢	国内服务器配阿里云镜像加速器：修改 /etc/docker/daemon.json
Flask 容器起不来，docker logs 显示连接 MySQL 失败	容器内无法访问云 MySQL 内网地址；.env 文件没正确加载	1. ECS 到云 MySQL 网络是否通 2. --env-file 路径是否正确 3. docker exec 进容器手动测 MySQL 连通性
Nginx 容器报 host not found: "flask"	两个容器不在同一个 Docker 网络	确认 docker-compose.yml 中没有自定义网络导致隔离，确保两服务在同一 compose 文件中
访问网站 502 Bad Gateway	Nginx 容器可以启动但连不上 Flask 容器	1. Nginx 的 proxy_pass 是否写成了 http://flask:5000（不是 127.0.0.1）2. Flask 容器的 5000 端口是否正确暴露在内部网络
docker compose up 后端口冲突	宿主机 80 端口被占（之前的 Nginx 还在跑）	先 systemctl stop nginx（关掉宿主机 Nginx），再启动 Docker 版 Nginx
代码改了但网站没变	容器里跑的还是旧镜像	重新 docker build + docker compose up -d（不是 restart，restart 不会用新镜像）
.env 文件不小心提交到 Git	Docker 的 .env 包含数据库密码	创建 .gitignore，添加 .env；如果已经提交了，换数据库密码作为补救
