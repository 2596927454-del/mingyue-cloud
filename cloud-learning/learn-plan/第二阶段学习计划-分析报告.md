第二阶段学习计划
前后端分离 + 反向代理 + 数据库
2026年5月 | 路线图阶段：第二阶段（第3~4周） | 难度：初级


一、问题说明
1.1 问题描述
第一阶段完成后，你已经拥有一个能通过域名访问的静态个人简历页。现在需要在此基础上添加留言板功能——访客可以在页面底部填写昵称和留言内容，提交后数据存入云端数据库，所有访客都能看到留言列表。
与第一阶段不同：第一阶段只涉及"静态文件托管"，一条 Nginx 指令就搞定了。第二阶段引入了一个真正的后端程序（Python Flask），以及数据库（SQLite）。这意味着你需要理解前后端分离、反向代理、REST API 这三个新概念。
1.2 学习背景与意义
第二阶段是你第一次跑通"前端 → Nginx 反向代理 → 后端 → 数据库"这条完整链路。这也是生产环境中绝大多数 Web 应用的标准架构。
留言板功能本身不复杂（就两个 API 接口 + 一张数据库表），但它的价值在于：
反向代理的自然场景：Flask 跑在 5000 端口，用户不可能记端口号，你必须配 Nginx proxy_pass 把 /api/ 请求转发过去——正好练刚学的 Nginx 配置
前后端分离的第一次实践：HTML 是静的，数据是动的，两者靠 JavaScript fetch API 对接。你会在同一页面体验到"静态渲染"和"动态数据加载"的区别
数据库第一次亮相：留言不再存储在 HTML 文件里，而是存在 SQLite 数据库文件中。每一次留言提交，都经历"前端 → Nginx → Flask → SQL → 磁盘"的完整数据流
Linux 技能的深化：不再只是装 Nginx，还要配虚拟环境、管进程、查端口、看日志、排查 502/500 错误
1.3 涉及的核心概念
以下是本阶段需要理解的核心概念，按学习顺序排列：
概念	在本阶段你需要知道的程度
Flask	一个轻量级的 Python Web 框架。本阶段只需要知道：用 @app.route 定义接口、用 jsonify 返回 JSON、用 request.get_json 接收数据
SQLite	一个文件型数据库，不需要安装和配置，一个 .db 文件就是整个数据库。本阶段只需要知道建表（CREATE TABLE）和增删查（INSERT/SELECT/DELETE）
REST API	前后端通信的约定格式。GET 用于获取数据，POST 用于新增数据。数据以 JSON 格式在前后端之间传递
反向代理 (Reverse Proxy)	Nginx 把特定路径的请求转发给后端服务的机制。浏览器访问 :80 的 /api/messages，Nginx 帮你转给 :5000 的 Flask，浏览器完全不知道 Flask 的存在
JavaScript fetch API	浏览器内置的 HTTP 请求工具。前端通过它向后端发 GET/POST 请求，拿到 JSON 数据后用 JS 动态渲染到页面上
虚拟环境 (venv)	Python 项目的"隔离沙箱"，每个项目用自己独立的包，互不干扰。Ubuntu 24.04 强制要求使用虚拟环境安装 pip 包
进程管理	后台运行 Python 程序、查看端口占用、杀掉残留进程等 Linux 运维技能
二、解决方式
2.1 方案概述
总体思路：在简历 HTML 底部嵌入留言板前端（表单 + 列表），用 Flask 写后端提供两个 API 接口，Nginx 配置反向代理把 /api/ 请求转发给 Flask，数据库用 SQLite（零配置）。最后申请 Let's Encrypt 免费 SSL 证书，给站点加上 HTTPS。

架构图：
浏览器 (你的 HTML 页面)
  │
  ├─ GET /index.html ──▶ Nginx :80 ──▶ 直接返回静态文件 (简历 + 留言板前端)
  │
  └─ GET/POST /api/messages ──▶ Nginx :80 ──▶ proxy_pass ──▶ Flask :5000 ──▶ SQLite (messages.db)
2.2 详细步骤
第 1 天：本地搭建 Flask 后端 + SQLite
创建项目目录 resume/，初始化 app.py
用 flask 库创建最简单的 Web 应用，定义两个路由：GET /api/messages（返回留言列表 JSON）、POST /api/messages（接收并存储留言）
用 Python 内置的 sqlite3 模块操作数据库：init_db() 建表，INSERT 写入留言，SELECT 读取留言
本地启动 Flask（默认监听 127.0.0.1:5000），用 curl 测试两个接口
验收：curl http://127.0.0.1:5000/api/messages 返回 []（空数组），POST 一条留言后再 GET 返回包含该留言的 JSON 数组

第 2 天：前端留言板嵌入简历页
在 index.html 底部添加留言板区域：昵称输入框 + 留言文本框 + 提交按钮 + 留言列表展示区
用 JavaScript fetch API 调用 /api/messages：页面加载时执行 GET 请求获取留言列表，表单提交时执行 POST 请求发送留言数据
处理三种 UI 状态：加载中（loading）、空数据（"还没有留言"）、错误（"加载失败"）
做基本的 XSS 防护：用 escapeHtml() 函数把用户输入中的 HTML 标签转义
验收：本地同时启动 Flask 和用浏览器打开 index.html，能提交留言并看到列表

第 3 天：Nginx 反向代理配置
理解反向代理的作用：浏览器不知道 Flask 在 5000 端口，它只和 Nginx 的 80 端口通信。Nginx 根据 URL 路径决定：/ 开头的直接返回静态文件，/api/ 开头的转发给 Flask
在 Nginx 配置中添加 location /api/ 反向代理规则：proxy_pass http://127.0.0.1:5000;
理解 Nginx 配置的层级结构：http {} → server {} → location {}，server 块必须放在 http 里面
理解 sites-available / sites-enabled 的标准管理模式
验收：curl http://localhost/api/messages 能正常返回 JSON（经过 Nginx 转发的）

第 4 天：部署到云服务器 + 踩坑实战
用 scp 把 index.html、app.py、requirements.txt 上传到服务器
在服务器上创建 Python 虚拟环境（python3 -m venv venv），激活后安装 flask
启动 Flask 后台运行（python app.py &）
配置 Nginx 站点（sites-available/mingyue → sites-enabled/），删除默认 default 站点避免冲突
学会排查部署问题的分层方法：curl 127.0.0.1:5000 确认后端正常 → curl localhost/api/messages 确认 Nginx 转发正常 → 浏览器测试
验收：通过域名访问个人网站，留言板功能完全正常

第 5 天：配置 HTTPS（Let's Encrypt 免费 SSL 证书）
安装 certbot + Nginx 插件：apt install certbot python3-certbot-nginx -y
执行 certbot --nginx -d 你的域名，certbot 会自动修改 Nginx 配置添加 443 端口监听和证书路径
certbot 自动配置 HTTP → HTTPS 跳转和证书续签定时任务
验收：浏览器访问 https://你的域名，地址栏显示小锁图标
2.3 核心原理分析
当你提交一条留言时，背后发生了什么？
完整链路如下：
表单点提交 → JS 阻止默认提交 → JSON.stringify() 序列化数据 → fetch POST ▎ /api/messages → Nginx 最长前缀匹配 /或/api/ → 匹配/返回静态网页，匹配/api转发 Flask → 路由匹配  ▎ 请求匹配相应接口和函数，函数解析请求体 → 验证数据 → 取北京时间 → sqlite3 INSERT → WAL ▎ 先写日志再写主文件 → Flask 返回 HTTP 状态码 + 对应的 JSON 响应体 → Nginx 传回浏览器 → ▎ 清空表单 + Toast 提示 → fetch GET /api/messages 再过一遍完整链路（SELECT ▎ 查库） → JSON 数组返回 → renderMessages() 渲染留言卡片

1. 前端发起请求
你在表单填入昵称和内容，点击"留下足迹"。JavaScript 阻止了表单的默认提交行为（e.preventDefault()），用 fetch API 发送一个 POST 请求到 /api/messages，请求体是 JSON 格式的 {nickname: "明月", content: "你好"}。

2. Nginx 反向代理
请求到达服务器的 80 端口，Nginx 正在监听。Nginx 检查请求路径 /api/messages，发现它匹配 location /api/ 规则，于是把整个请求原封不动地转发给 http://127.0.0.1:5000。浏览器完全不知道这个转发过程——它以为自己一直在和同一个服务器通信。

3. Flask 处理请求
Flask 收到请求，根据路由 @app.route("/api/messages", methods=["POST"]) 匹配到 create_message 函数。函数解析 JSON 请求体，验证昵称和内容不为空，把当前时间格式化为北京时间字符串，然后执行 INSERT INTO messages 写入数据库。

4. SQLite 写入磁盘
sqlite3 模块把 INSERT 语句翻译成对 messages.db 文件的写入操作。SQLite 的 WAL（Write-Ahead Logging）模式先写日志再写主文件，保证了写入的原子性和并发安全。写入成功后，数据就永久存储在了 messages.db 文件里。

5. 返回响应
Flask 返回 HTTP 201 Created 状态码 + {"ok": true} 的 JSON 响应体。Nginx 把响应传回浏览器。前端收到响应后，清空表单、显示"留言成功"提示、重新调用 GET /api/messages 刷新留言列表。

为什么用反向代理而不是直接暴露 Flask？
安全性：Flask 监听 127.0.0.1（只有服务器本机可达），外部完全无法直接访问。攻击者碰不到 Flask，只能通过 Nginx
端口统一：用户只需要记 80 端口（HTTP）或 443 端口（HTTPS），不需要知道 Flask 在哪个端口
静态文件性能：Nginx 是 C 语言写的高性能服务器，处理静态文件比 Python Flask 快得多。让 Nginx 做它擅长的事，Flask 只处理 API 逻辑
扩展性：未来如果有多个后端服务（比如 /api/payment、/api/chat），只需要在 Nginx 里加新的 location 规则，前端完全不受影响

数据库为什么用 SQLite 而不是 MySQL/PostgreSQL？
SQLite 是一个文件型数据库，不需要安装数据库服务、不需要配置用户密码、不需要管理进程。整个数据库就是一个 messages.db 文件，和 index.html、app.py 放在一起。对于留言板这种低并发场景，SQLite 绰绰有余。这让你先把"前后端分离 + 反向代理 + API"的主链路跑通，第三阶段再引入云数据库（MySQL/PostgreSQL）。

2.5 预计踩坑与应对
现象	可能原因	排查方法
pip install flask 报错 externally-managed-environment	Ubuntu 24.04 禁止全局 pip 安装	创建虚拟环境：python3 -m venv venv && source venv/bin/activate
端口 5000 被占用 (Address already in use)	上次 python app.py & 的后台进程没杀	kill $(lsof -t -i:5000) 或 ss -tlnp | grep 5000 查 PID
nginx -t 报错 "server" directive is not allowed here	server 块写到了 http {} 大括号外面	检查 http {} 的闭合位置，把 server 移进去；或写到 sites-available 下
curl Flask 5000 正常但网页留言报错	Nginx 的 sites-enabled/default 抢了 80 端口，没有 /api/ 规则	rm /etc/nginx/sites-enabled/default，把自己的站点链过去
curl 5000 卡住没响应	Flask debug 模式双进程残留，或 WAL 锁文件没清理	kill -9 所有 python PID，删掉 messages.db-shm / .db-wal，重启 Flask
网页留言"加载失败"	Nginx /api/ 转发没配或配错了	先 curl localhost/api/messages 确认 Nginx 层是否通
Flask 返回 502 Bad Gateway	Flask 进程没启动或挂了	ss -tlnp | grep 5000 确认端口是否在监听
上传文件后找不到位置	scp 传到了错误的路径	pwd 确认当前目录，ls 确认文件存在，Nginx root 路径和文件实际路径要一致
